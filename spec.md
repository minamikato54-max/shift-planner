# 要件定義書 — Shift Planner（シフト希望提出・自動割当・通知アプリ）

## 誰の・何を・どう解決するか

- **誰の**: 1つの職場（アルバイト・パート先など）で働く複数人のスタッフと、シフトを組む管理者
- **何を**: 管理者が「希望提出期間」とスロット（朝/昼/夜など必要人数付きの時間帯枠）を作成 → スタッフが日付ごとに勤務可能な時間帯を入力 → 管理者が自動生成された下書きシフトを確認・調整して確定 → 確定シフトを各スタッフにGmailまたはLINEで自動通知する、一気通貫のアプリ
- **どう解決するか**: 紙・LINE・口頭でバラバラに集めていたシフト希望の「収集」「割り当て」「連絡」の3工程をすべて1つのアプリに集約し、管理者の手作業（集計・突き合わせ・個別連絡）を最小化する

## 想定規模・ロール

- 1職場・スタッフ数人〜十数人規模を想定（複数職場のマルチテナント化は今回スコープ外）
- 2ロール:
  - **スタッフ**: 自分の希望を提出・修正、通知チャネルの設定ができる
  - **管理者**: 募集期間・スロットの作成、自動割当の実行・調整・確定、通知の送信ができる（スタッフ操作もできる）
- 権限付与はFirebase Consoleで`users`コレクションの`role`フィールドを手動編集する運用とする（アプリ内に権限付与UIは作らない）。初回ログイン時は自動的に`role: "staff"`で作成される

## コア機能（MVP）

### 希望収集

1. Googleログイン（初回ログイン時に`users/{uid}`ドキュメントを`role: "staff"`, `notifyChannel: "gmail"`で自動作成）
2. 【管理者】募集期間の作成（タイトル・開始日・終了日・スロットテンプレートを指定して都度作成）
   - スロットテンプレート例: `朝 9:00-13:00 必要2人` `昼 13:00-18:00 必要2人` `夜 18:00-22:00 必要1人`（期間内の全日に同一パターンを適用。曜日ごとの違いは今回スコープ外）
3. 【管理者】募集期間の一覧表示
4. 【スタッフ】現在募集中の期間一覧から1つ選び、期間内の日付ごとに以下のいずれかを入力
   - 勤務可能な時間帯（開始時刻・終了時刻）
   - 出勤不可（NG）
5. 【スタッフ】提出済みの内容はいつでも再編集して上書き保存できる（締切による自動ロックは今回なし）
6. 【管理者】募集期間を1つ選ぶと、その期間の全スタッフ分の提出内容を一覧表（行=スタッフ、列=日付、セル=時間帯 or NG）で確認できる

### 自動割当

7. 【管理者】「下書き自動生成」ボタンで、期間内の各日×各スロットに対して条件を満たすスタッフを自動選出する
8. 【管理者】生成された下書きシフトを画面上で確認し、割り当ての追加・削除・入れ替え（手動調整）ができる
9. 【管理者】不足（`requiredCount`を満たせなかったスロット）は一覧上で視認できる
10. 【管理者】内容を確認後「確定」すると割当が確定し、通知が送信される

### 通知

11. 【スタッフ】プロフィール画面で通知チャネル（Gmail / LINE）を選択できる
12. 【スタッフ】LINEを選ぶ場合、専用の公式アカウントを友だち追加 → メッセージ送信で返ってきた`userId`をプロフィール画面に貼り付けて連携する
13. シフト確定時、各スタッフに「自分の担当分（日付・スロット名・時間帯の一覧）」を選択済みチャネルで自動送信する
14. 【管理者】送信結果（成功/失敗）を確認できる（未連携・不達などの失敗を一覧で把握できる）

### 確定シフトの共有閲覧

15. 【全員（スタッフ／管理者）】確定済みの募集期間を選ぶと、その期間の全スタッフ分の割当結果（日付×スロット×担当者名）を一覧で閲覧できる。個別通知（Gmail/LINE）はあくまで「自分の担当分だけを知らせる」ためのもので、これとは別に「部門全体の確定シフトを一覧で見る」ニーズに応える画面。未確定（下書き）の期間は管理者以外には表示しない

## 自動割当ロジック仕様

- **対象条件**: その日の`availabilities.entries[date].status === "ok"` かつ 申告時間帯がスロットを完全に包含（`entry.start <= slot.start && entry.end >= slot.end`）
- **選出ロジック**: 条件を満たす候補者のうち、その期間内での累計割当数が少ない人から優先して`requiredCount`人を選ぶ（簡易的な負荷分散。同数の場合は登録順）。同一スタッフが同日の重複時間帯のスロットに二重登録されないようにする
- **不足時の扱い**: 候補者が`requiredCount`に満たない場合は自動で欠員募集はせず、「不足◯人」として一覧に表示するのみ。埋めるかどうかは管理者の手動調整に委ねる
- **意図的に単純化している点**: 各スタッフの希望優先度・連続勤務日数の考慮・複雑な数理最適化は行わない。まずは「機械的に埋める→人が最終調整する」を優先する

## 通知機能仕様

- Gmail送信は開発者自身のGmailアカウント1つをNodemailer（SMTP、アプリパスワード）の送信元として使い、各スタッフの`email`宛に送信する（スタッフ個別のGmail API認可は行わない）
- LINE送信は本アプリ専用に新規作成するLINE公式アカウント（Messaging API。health-botのチャネルとは別物、既存設定には触れない）を使う
  - 連携フロー: スタッフが公式アカウントを友だち追加 → 何かメッセージを送信 → Webhookが`userId`を自動返信 → スタッフがそれをコピーしてプロフィール画面の入力欄に貼り付け保存
  - OAuth（LINEログイン）は使わない。実装・設定の手間を減らすための意図的な簡略化
- 送信失敗（LINE未連携、メールアドレス不備など）はエラーとして記録し、管理者が「送信できなかった人」を確認できるようにする

## スコープ外（今回はやらない）

- 曜日ごとに異なるスロットパターン（期間内は同一パターン固定）
- 各スタッフの希望優先度・連続勤務制限などを考慮した高度な最適化（シンプルな負荷分散ルールのみ）
- シフト交換・欠勤代打の募集機能
- 提出リマインド通知（希望提出前の催促）
- 複数職場・複数店舗のマルチテナント対応
- 提出期限による自動締切／編集ロック
- アプリ内での管理者権限付与UI（Firebase Console手動運用とする）
- LINEログイン（OAuth）による自動連携

## 技術スタック

- Next.js (App Router) + TypeScript + Tailwind CSS
- Firebase Authentication（Googleログイン）
- Firebase Firestore
- Nodemailer（Gmail SMTP、アプリパスワード）
- LINE Messaging API（本アプリ専用の新規チャネル）
- Vercel デプロイ

## データモデル（案）

### `users` コレクション（ドキュメントID = uid）

| フィールド    | 型        | 説明                                                                           |
| ------------- | --------- | ------------------------------------------------------------------------------ |
| uid           | string    | Firebase AuthのUID（ドキュメントID）                                           |
| displayName   | string    | 表示名                                                                         |
| email         | string    | メールアドレス                                                                 |
| role          | string    | `"staff"` または `"admin"`（初期値は`"staff"`、管理者権限はConsoleで手動変更） |
| notifyChannel | string    | `"gmail"` または `"line"`（デフォルト`"gmail"`）                               |
| lineUserId    | string?   | LINE連携済みの場合のuserId（未連携時はnull）                                   |
| createdAt     | timestamp | 初回ログイン日時                                                               |

### `periods` コレクション（募集期間）

| フィールド | 型        | 説明                                                                                        |
| ---------- | --------- | ------------------------------------------------------------------------------------------- |
| id         | string    | ドキュメントID                                                                              |
| title      | string    | 期間の名前（例:「8月前半」）                                                                |
| startDate  | string    | 開始日（`YYYY-MM-DD`）                                                                      |
| endDate    | string    | 終了日（`YYYY-MM-DD`）                                                                      |
| slots      | array     | `[{ name: "朝", start: "09:00", end: "13:00", requiredCount: 2 }, ...]`（期間内の全日共通） |
| createdBy  | string    | 作成した管理者のuid                                                                         |
| createdAt  | timestamp | 作成日時                                                                                    |

### `availabilities` コレクション（希望提出、1ユーザー×1期間で1ドキュメント）

| フィールド | 型        | 説明                                                                                                      |
| ---------- | --------- | --------------------------------------------------------------------------------------------------------- |
| id         | string    | `${periodId}_${uid}` 形式の複合ID                                                                         |
| periodId   | string    | 対象の募集期間ID                                                                                          |
| userId     | string    | スタッフのuid                                                                                             |
| userName   | string    | スタッフの表示名（一覧表示用にスナップショットとして保持）                                                |
| entries    | map       | `{ "2026-08-10": { status: "ok", start: "09:00", end: "17:00" }, "2026-08-11": { status: "ng" } }` の形式 |
| updatedAt  | timestamp | 最終更新日時                                                                                              |

### `assignments` コレクション（自動割当の下書き・確定シフト、1期間=1ドキュメント）

| フィールド    | 型        | 説明                                                                                                                                                                                                       |
| ------------- | --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| id            | string    | `periodId`と同一の値（1期間につき1ドキュメント。取得は常にIDで直接行い、クエリは使わない）                                                                                                                 |
| periodId      | string    | 対象の募集期間ID                                                                                                                                                                                           |
| byDate        | map       | `{ "2026-08-10": { "朝": ["uid1","uid2"], "昼": ["uid3"], "夜": [] } }`                                                                                                                                    |
| staffNames    | map       | `{ "uid1": "山田太郎", "uid2": "佐藤花子" }`（`availabilities.userName`のスナップショット。全員閲覧画面で名前表示するために保持。`users`コレクションを他人が横断参照しなくて済むようにするための非正規化） |
| confirmed     | boolean   | 確定済みかどうか                                                                                                                                                                                           |
| confirmedAt   | timestamp | 確定日時（未確定はnull）                                                                                                                                                                                   |
| notifyResults | map       | `{ "uid1": "sent", "uid2": "failed" }`（確定後の送信結果、未送信時は存在しない）                                                                                                                           |

**Firestoreクエリ設計の注意（nomi-checkでの詰まりポイントを踏襲）**: `availabilities`を管理者が一覧取得する際は `where("periodId", "==", ...)` の単一条件のみに留め、`orderBy`と組み合わせた複合クエリにしない。ソート・整形は取得後にクライアント側（JS）で行う。`assignments`はそもそも`periodId`をドキュメントIDとして使うため、クエリ自体が不要（`getDoc`で直接取得）。これにより未作成の複合インデックスによるサイレント障害（一覧が空に見える等）を避ける。

## 画面構成（概要）

1. ログイン画面（Google認証）
2. ホーム画面（ロールに応じて表示切り替え。スタッフ用メニュー＋管理者には管理者メニューも追加表示）
3. 【スタッフ】希望提出画面（募集期間を選択→日付ごとに時間帯 or NGを入力）
4. 【スタッフ】プロフィール/通知設定画面（通知チャネル選択、LINE連携用userId入力欄）
5. 【管理者】募集期間作成画面（タイトル・期間・スロットテンプレート入力）
6. 【管理者】提出状況一覧画面（期間を選択→スタッフ×日付のテーブル表示）
7. 【管理者】下書きシフト画面（自動生成ボタン→日付×スロットの割当結果表示→手動調整→確定ボタン→送信結果表示）
8. 【全員】確定シフト閲覧画面（確定済みの募集期間を選択→日付×スロット×担当者名の一覧表示。読み取り専用、編集不可）

## 非機能要件

- Firestoreセキュリティルールで、`availabilities`は本人（`userId`一致）または`role == "admin"`のみ書き込み/閲覧可能に制限。`periods`の作成・更新は`role == "admin"`のみ。`assignments`は書き込みが`role == "admin"`のみ、読み取りは「`role == "admin"`」または「`confirmed == true`（確定済みの場合のみ、認証済みの全ユーザーが閲覧可）」
- レスポンシブ対応。ただし管理者の一覧表・下書きシフト画面はPC想定で横スクロール対応にする
- LINE Messaging APIのチャネルアクセストークン、Gmail送信用のアプリパスワードは環境変数（`.env.local`、Vercelの環境変数）で管理し、コミットしない
- デプロイ後チェックリスト（nomi-checkでの詰まりを踏まえ、事前に対応する）:
  - Firebase Authenticationの「Google」プロバイダを**有効化トグルON**にしてから動作確認する
  - Vercel本番ドメインをFirebase Authenticationの「承認済みドメイン」に追加してから本番確認する
  - Firestoreクエリは複合インデックスが必要な設計にしない（上記データモデルの注意を参照）
  - LINE公式アカウント作成時、Messaging APIのWebhook URLをLINE Developersコンソールに登録し、Verify疎通確認を行う（新規の詰まりポイントとして事前に手順化しておく）
  - Gmail送信用アカウントで「アプリパスワード」を発行する際は2段階認証の有効化が前提になる点に注意する
